对于gc的优化，最好的结果就是不影响用户线程的执行，没有任何停顿，即Throughput 为100%，Latency为0。
但目前所有的垃圾收集器都做不到。

决定JVM性能的属性：1. 吞吐量（Throughput ）；2. 延时（latency）；之前听陈涛（《Hotspot实战》的作者）分享jvm故障诊断，他提到吞吐率>99%、full gc 次数小于1算正常；

搜索业务要求低停顿，CMS是以获取最短回收停顿时间为目标的收集器。

G1也是低延迟的收集器，为啥不选择它？

plumbr公司的测试结论（2013年的测试）：
https://plumbr.eu/blog/garbage-collection/g1-vs-cms-vs-parallel-gc

If I would dare to make such a conclusion, I would say that CMS is still the best “default” option to go with. G1 throughput is still so much worse that the improved latency is usually not worth it.

推荐使用G1的场景

G1垃圾收集器集中几种的应用场景是在应用程序有大内存并且要求极小的延迟的情况下，这意味着堆内存应该至少6G以上，稳定的可预测的停顿时间在0.5秒以下。

如果在使用CMS或者ParallelOldGC 垃圾收集器出现以下的情况时，切换G1垃圾收集器可以带来更好的效果：

（1）Full GC 太长或者太频繁
（2）对象的分配内存空间率或者晋升频率差异很大
（3）追求更快的响应时间，当停顿时间大于0.5秒之1秒时

注意：如果你使用CMS或者ParallelOldGC 的gc时间可以接受，最好的方式仍然使用当前的GC垃圾收集器，
此外改为G1垃圾收集器，并不意味着，你就要一定升级到最新的JDK版本，当然如果可以的话，还是尽量使用当然最新最稳定的版本。

结合plumbr的结论，和搜索应用内存使用情况（小于6G），我们暂时选择CMS来作为搜索系统的gc收集器。
